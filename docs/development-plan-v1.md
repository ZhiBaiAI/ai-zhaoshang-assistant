# V1 开发计划

## 阶段 0：项目整理

目标：把现有抖音 MVP 纳入正式项目结构。

- 建立项目目录
- 迁移 `douyin-agent`
- 补齐产品、架构、部署、开发计划文档
- 确认 V1 技术栈

## 阶段 1：抖音 Agent 稳定化

目标：让抖音自动化具备稳定读写能力。

- 保持登录态
- 读取会话列表
- 识别新消息
- 消息去重
- 发送回复
- 发送成功/失败记录
- 登录失效检测
- 页面异常截图
- 人工接管开关

## 阶段 2：基础设施

目标：搭建正式数据和任务底座。

- PostgreSQL + pgvector
- Redis + BullMQ
- 数据模型和迁移
- 项目配置表
- 会话表
- 消息表
- 回复任务表
- 线索表
- 同步和通知日志表

当前已完成基础 SQL 和内存/PostgreSQL store：

- 项目配置表
- 会话和消息入库
- 新私信自动创建回复任务
- 线索表
- 知识库表
- 操作日志表

## 阶段 3：知识库和 RAG

目标：基于本地资料生成可控回复。

- 项目资料模板
- Docling / Unstructured 文档解析适配
- LangChainJS 文档切片和入库编排
- 本地 embedding 模型封装
- PostgreSQL + pgvector 写入
- PostgreSQL tsvector 关键词索引
- FAQ 匹配
- 招商规则结构化存储
- 关键词检索
- 向量检索
- 混合召回合并和评分
- 检索结果组装
- 知识库更新流程
- 切片预览和问答测试台预留

技术决策见 `docs/knowledge-rag-v1.md`。

## 阶段 4：AI 回复引擎

目标：完成招商客服自动接待能力。

- 意图识别
- 多轮上下文摘要
- 回复生成 prompt
- 敏感内容校验
- 不确定问题转人工
- 留资引导
- 回复日志
- Provider 可切换

当前已完成基础实现：

- OpenAI-compatible LLM provider
- 回复任务生成建议回复
- `auto` 模式生成后自动加入发送队列
- `assisted` 模式支持人工审核后加入发送队列
- RAG 上下文组装
- 风险问题转人工

## 阶段 5：留资和线索

目标：把咨询沉淀为销售可跟进线索。

- 手机号识别
- 微信号识别
- 城市识别
- 预算和开店时间提取
- 会话摘要
- 意向等级判断
- 自动创建线索
- 自动更新线索
- 跟进状态模型

当前已完成基础实现：

- 手机号、微信、城市、预算、开店时间提取
- 高意向线索自动创建/更新

## 阶段 6：飞书和通知

目标：让销售直接用起来。

- 飞书多维表格模板
- 新线索同步
- 线索更新同步
- 飞书状态同步回本地
- 企业微信机器人
- 钉钉机器人
- 飞书机器人
- 高意向和异常通知

V1 先只实现飞书：

- 飞书开放平台 client
- 飞书机器人 webhook 通知
- 飞书开放平台 IM 文本消息预留
- 高意向线索同步到飞书多维表格
- 飞书多维表格线索状态反向同步
- 管理台查看和处理项目、会话、回复任务、线索、日志

## 阶段 7：试运行和交付

目标：用真实抖音账号跑通闭环。

- 只读模式
- 半自动模式
- 小范围自动回复
- 回复质量测试
- 留资识别测试
- 飞书同步测试
- 通知测试
- 登录失效和异常演练
- 交付文档和客户培训

## 阶段 8：评价渠道扩展

目标：支持大众点评等评价平台的评论获取、分析报表和自动回评。

当前已完成基础框架：

- 通用评价事件类型和 `dianping` 来源
- 评价入库和去重
- 评价回复任务生成
- 评价回复建议生成和发送队列
- 评价回复发送结果回写
- 评价日报和月报生成
- `dianping-agent` 样例文件导入和 API 上报
- 管理台评价分析入口

## 优先级

第一优先级：

- 抖音读写稳定性
- 消息去重
- 人工接管
- 会话和消息入库

第二优先级：

- 本地知识库
- AI 回复
- 留资识别
- 线索入库

第三优先级：

- 飞书同步
- 销售通知
- 部署和运维监控
